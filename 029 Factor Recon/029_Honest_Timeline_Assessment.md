## Section 1 — Planned vs. Realistic Timeline

### 1.1 Planned Timeline

**Committed Plan:** 4 sprints × 2 weeks = **8 weeks**, **776 hours**, starting April 8, 2026.

**Why it is achievable:**

1. **160h of reuse savings front-loaded into Sprint 1.** All four platform libraries (Logging, LoginAgent, Environment, Excel.Utilities) and all REFramework/Common/WorkQueue components are 0-effort imports from 058 and 057. This frees Sprint 1 to deliver substantial library build work alongside foundation setup.

2. **Team allocation is maximized from Day 1.** No ramp-up, no Sprint 0. DEV-1 and DEV-2 run at 80% from Week 1. LEAD runs at 80% (64h available; ~52h planned work; ~12h contingency buffer) as active developer + reviewer throughout.

3. **Agent handles all documentation in Sprint 1 and Sprint 4.** BRD, TDD, RC, Plan, Dev Guide generated by AGENT at sprint start — developers receive full specs before first Studio session.

4. **MCH Library build follows a clear pattern.** BGMM, BGIR, BSEC, BPOP are read-only extractions. PYDN/PYUP/PYDA are posting variations that share the same structural pattern (navigate to screen → enter row data → confirm). Once one posting screen is built in Sprint 2, the next two are 60% adaption effort.

5. **PYDN → PYUP → PYDA and UMAS variants: parallel pattern reuse within sprint.** DEV-1 builds PYDN; DEV-2 adapts PYUP and PYDA simultaneously using the same structure.

## Section 4 — What Will Definitely Take Longer Than Expected

1. **MCH Screen Selector Capture (Steps 3 + 8 — Sprints 1–3, BGMM through PYDA).**
   Planned: 2h per screen (part of S-M estimates). Realistic: 4–6h per screen for first 3 screens, then 2–3h with pattern established. Reason: AVIVA terminal selectors are notoriously unstable — they depend on terminal font, window size, zoom level, and session state. The BGMM screen alone may have 20+ fields requiring individual selector validation.
   **Plan for 12h of extra selector work across all MCH screen workflows.**

2. **UMAS DLL Integration (S3-08, Sprint 3).**
   Planned: 24h for 3 UMAS variants. Realistic: 32–40h if DLL requires Custom Activity wrapper or if UMAS method signatures differ from BP Object logic. Reason: DLL COM interop in unattended Orchestrator context requires elevation configuration, DLL registration, and process isolation that BP handles natively but UiPath requires explicit setup.
   **Plan for UMAS to consume S3-08's full 24h plus overflow into Sprint 4 buffer. Descope trigger: 20h into S3-08.**

3. **Bloomberg HTTP WS Integration (S2-07, Sprint 2).**
   Planned: 20h. Realistic: 24–28h. Reason: Bloomberg APIs vary by product tier and authentication model — the BP Object uses HTTP GET but the actual auth mechanism (API key in header? session cookie? OAuth?) is not documented in the .bprelease. Discovering and implementing the correct auth pattern adds 4–8h.
   **Plan for 28h allocation; get Bloomberg API docs from integration team before Sprint 2 Week 1 starts.**

4. **DataTransformation Business Logic Verification (S3-02/S3-03, Sprint 3).**
   Planned: 26h. Realistic: 32–36h. Reason: The BP Object actions Filter_Calculate, Set_Column_Data, and Change_Column_Position hide complex LINQ and VB.Net inline logic that is not visible from the .bprelease XML subsheet names alone. A 2-hour walkthrough with the GIO SME is required to confirm all transformation rules before code is written.
   **Plan for a Sprint 3 Week 5 SME session (2h) before S3-02 starts. This time is included in the estimate.**

5. **UAT — Business Comparison with BP Baseline (S4-08, Sprint 4).**
   Planned: 12h for LEAD + 8h for DEV support. Realistic: 18–24h if BP output files are not immediately available for comparison. Reason: The BP system may still be processing live production COB data; obtaining 3 parallel BP baseline outputs requires coordination with RPA Support L1 and business operations. Timing mismatch with the COB window is a real constraint.
   **Plan for UAT to require 2 full COB days of parallel run, not 1. LEAD books the UAT window to align with COB schedule from Sprint 3 planning.**

## Section 5 — What Will Go Faster Than Expected

1. **REFramework + Config.xlsx — Day 1 productivity.** Because the REFramework template and Config.xlsx are cloned from 057, DEV-1 has a working runnable project from Hour 1 of Sprint 1. No blank-page setup. The 4 NuGet library imports add 3 additional hours of value before any custom code begins.

2. **Agent XAML scaffolding reduces DEV Studio time significantly.** For all 35+ workflows, Agent generates: argument definitions, Try/Catch wrappers, invocation structure, and VB.Net expressions. DEV's time in Studio is focused solely on adding real selectors, testing against live systems, and debugging — not creating boilerplate. This saves approximately 1–2h per workflow = ~30–50h across the project.

3. **PYDN → PYUP → PYDA posting pattern reuse within Sprint 2–3.** Once MCH_PYDN is correctly built and tested (est. 20h), MCH_PYUP is ~14h (same structure, different screen code) and MCH_PYDA is ~12h (same structure again). Within Posting Library: FR_PYDN_Posting done correctly → FR_PYUP and FR_PYDA are each ~70% of the first effort. Three standard postings delivers 3× value for ~2.2× effort.

4. **LEAD as active integration and quality driver in Sprint 3.** LEAD-1's 52h allocation is not passive review — it includes integration pre-testing, exception matrix validation, and rapid architecture decisions during build. This prevents review delays and catches quality issues in-sprint.

5. **Dispatcher processes are thin.** Once MainScheduler Dispatcher (S2-10) is built, Adhoc and NOVAL dispatchers are adapations of the same codebase — each is 4–6h of customization (not a full build). Complete Adhoc and NOVAL in Sprint 4 in 8h total vs. the 14h+ they would cost if built from scratch independently.

## Section 7 — Key Recommendations

1. **Book MCH test environment before Sprint 1 Day 1 (LEAD-1 action).** Every MCH screen workflow is blocked without test environment access. A 1-week delay here ripples through Sprint 2 MCH posting screens and into Sprint 3 Posting Library. Reference: R-01, M4 at risk.

2. **Confirm RDR Cloud Client DLL .NET compatibility in Sprint 1 Week 1 (DEV-2 action).** Run the DLL in an isolated .NET 6 console project before S2-08 is scheduled. If incompatible, escalate to integration team and begin REST fallback design in parallel. Do not wait until Sprint 2 to discover this. Reference: R-02, S2-08.

3. **Schedule GIO SME DataTransformation logic walkthrough for Sprint 3 Week 5 (LEAD-1 action).** Filter_Calculate and BGI_Loop logic is entirely invisible in the .bprelease XML. A 2-hour walkthrough before S3-02 will prevent a rebuild later. This single session is the highest-ROI activity in the project. Reference: R-09, S3-02.

4. **Establish S3-07/S3-08 descope boundary at 20h combined (LEAD-1 decision).** Posting Library UMAS variants are the highest-risk component. LEAD-1 must set a hard limit: if UMAS variants (S3-08) are not working at the 20h mark, descope them to a post-go-live story (S5-01) and proceed with standard PYDN/PYUP/PYDA posting. Business must pre-approve this descope trigger. Reference: R-07, Sprint 3 critical path.

5. **Book UAT window aligned with COB schedule from Sprint 3 (LEAD-1 + Business action).** UAT for a COB automation must run during actual COB windows (after market close). This means a specific evening appointment with business team. Book this in Sprint 3 Week 6, before Sprint 4 begins. Reference: R-10, S4-08.

6. **Submit CyberArk provisioning request in Sprint 1 (LEAD-1 action).** CyberArk changes typically take 5–10 business days through Platform Engineering. Submit in Sprint 1 Week 1 to ensure it is ready before Sprint 3 credential testing. Reference: R-08, S3-09 credential testing.

7. **Obtain real Master Scheduler file in Sprint 1 for holiday format validation (DEV-1 action).** Holiday date format assumptions (USIS: M/d/yyyy, EMEA: dd-MMM-yyyy, APAC: yyyy/MM/dd, IIS: dd/MM/yyyy) are derived from BP workflow descriptions. Validate against a real scheduler file before building S2-05. If formats differ, the 4 regional validators need correction. Reference: R-09, S2-05.
